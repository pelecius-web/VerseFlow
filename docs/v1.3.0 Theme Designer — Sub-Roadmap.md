# VerseFlow v1.3.0 Theme Designer — Sub-Roadmap

> **Stage:** v1.3.0 — ✅ Complete (May 24, 2026)
> **Roadmap Category:** Category 4 — Customization & Theming
> **Primary Goal:** Add a per-channel theme system, in-app Theme Designer, image backgrounds, custom font import, and a preset theme library — without regressing the v1.1.0 dual-channel display architecture, the v1.2.0 NDI capture pipeline, or the operator panel's stable visual language.
> **Status:** Planned → Phase 1 Complete (May 15) → Phase 2 Complete (May 19) → Phase 3 Complete (May 23) → Phase 4 Complete (May 24)
> **Implementation Style:** Additive only. Zero regression to v1.1.x and v1.2.x behavior. Operator panel chrome is isolated from per-channel theming.

> [!IMPORTANT]
> **Deviation from main ROADMAP:** `docs/ROADMAP.md` lines 278, 290, and 393 specify QML for the Theme Designer preview and display rendering. This sub-roadmap **rejects QML** in favor of an embedded `DisplayWidget` (extracted from `DisplayWindow`) for the following reasons: (1) two rendering engines guarantee visual drift between designer and live output, (2) the NDI 30fps `QWidget.grab()` pipeline was hardened across four phases and must not be destabilized by a parallel QtQuick scenegraph, (3) the entire codebase is 100% QtWidgets with zero QML infrastructure. The ROADMAP should be updated to reflect this decision at lines 278, 290, and 393.

---

## 1. Executive Summary

v1.3.0 turns VerseFlow's hardcoded theme into a fully editable theme system. The stage delivers four capabilities:

- **Per-channel theme application** — Main and Alt can use different themes simultaneously without bleeding into the operator panel's QSS.
- **In-app Theme Designer** — A three-panel UI (saved themes / live preview / property editor) that uses a real `DisplayWidget` instance (extracted from `DisplayWindow`) for pixel-identical preview.
- **Image backgrounds** — PNG, JPEG, and SVG backgrounds for both fullscreen and lower-third modes, with logo image support replacing the v1.1.0 placeholder rectangle.
- **Custom fonts** — `.ttf` / `.otf` import via `QFontDatabase` at app startup, referenced by family name in theme JSON.
- **Preset library** — 8–10 professionally tuned themes shipped with the application.

The critical implementation rule is:

> The Theme Designer's preview surface MUST be a `DisplayWidget(QWidget)` instance — the same class that `DisplayWindow` uses as its central widget. Never a separate QML scenegraph or a custom paint path. What the operator sees in the designer is exactly what the congregation sees on the display, byte-for-byte.

> **Thumbnail presentation note:** the shipped theme thumbnails are allowed to use thumbnail-only presentation treatments to make presets easier to identify and more professional at a glance. That is a deliberate deviation from the original theme design contract for the thumbnail cards only. The live display remains the fidelity source; thumbnails are an identification surface, not a pixel-perfect replay of live output.

Three things explicitly **deferred** to v1.3.x point releases:

- **MP4 video backgrounds** — Belongs in the downstream broadcast chain (OBS/vMix/ATEM). Adding `QtMultimedia` introduces codec, GPU, and NDI capture timing risk during a live service.
- **Slide transitions** — Phase 3 ships fade only.
- **`.vftheme` archive format** — Reserved for the point release that introduces font and asset bundling. Until then, themes stay as flat `.json` with a `schema_version` field.

## 1.1 Architectural Decision Record

**Scoring rubric:** Scores reflect rendering parity (40%), regression risk (30%), dependency cost (20%), schedule fit (10%).

### Decision 1 — Theme Designer Preview Surface

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Extracted `DisplayWidget(QWidget)` embedded in designer** | Extract ~432 lines of rendering logic from `DisplayWindow(QMainWindow)` into a standalone `DisplayWidget(QWidget)`. Embed directly in the designer's center panel. Uses the exact same `paintEvent`, `_render_lower_third`, and font-fitting pipeline as the live display. | **95/100** | **Selected** |
| `_preview_mode` flag on `DisplayWindow(QMainWindow)` | Add a boolean to skip window flags. Simpler, but `QMainWindow` auto-creates `menuBar()`/`statusBar()`, `QMainWindowLayout` adds internal margins, and `sizeHint()` defaults to 256×192 — all causing geometry mismatches when embedded in a parent widget. | 55/100 | Rejected |
| QtQuick / QML preview with `QQmlPropertyMap` | More flexible animations, but introduces a second rendering engine (scenegraph vs. `QWidget.paintEvent`), risks divergence from real output, adds `PyQt6.QtQuick`/`QtQml` dependencies, and runs alongside the 30fps NDI `grab()` loop. | 35/100 | Rejected |

**Why `DisplayWidget` extraction was selected:**

1. **Rendering parity is non-negotiable.** A theme designer's value proposition is "what you see is what the congregation gets." The extracted widget IS the live rendering code — zero drift possible.
2. **`QMainWindow` cannot be cleanly embedded.** It auto-creates `menuBar()` and `statusBar()`, `QMainWindowLayout` reserves space for them even when hidden, and `sizeHint()` defaults to 256×192. These cause geometry mismatches in a `QSplitter` or parent layout.
3. **NDI capture must be protected.** Running QtQuick alongside the hardened 30fps `QWidget.grab()` loop introduces frame-timing collisions and overhead that v1.2.0 spent four phases eliminating.
4. **Infrastructure already exists.** The rendering methods (`_render_fullscreen`, `_render_lower_third`, `paintEvent`, font-fitting, logo, `QStackedWidget` pages) are self-contained within `DisplayWindow`. Extracting them into `DisplayWidget(QWidget)` is a mechanical refactor — `DisplayWindow` becomes a thin `QMainWindow` shell wrapping `DisplayWidget` as its central widget.

**Extraction scope:** ~432 lines across 7 methods: `_build_lower_third_page` (~116), `_render_lower_third` (~45), `_render_fullscreen` (~69), `_fit_lower_third_fonts` (~81), `_fit_church_name_font` (~56), `_update_lower_third_geometry` (~30), `paintEvent` (~35).

**Long-term consequence:** All future visual features (transitions, animations, advanced overlays) ship through `DisplayWidget.paintEvent` and the QStackedWidget pages, never through a parallel render engine.

### Decision 2 — Video Backgrounds: Defer

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Image backgrounds only (PNG, JPEG, SVG)** | Static backgrounds + downstream video chain (OBS/vMix). Covers ~95% of real church use. Zero new dependencies. | **90/100** | **Selected** |
| Image + MP4 video via `QtMultimedia` | Adds `QtMultimedia` + `QtMultimediaWidgets`, codec pack dependencies, GPU blacklisting risk, and frame-timing concerns in the `grab()` loop. | 38/100 | Deferred to v1.3.x |

**Why image-only was selected:**

1. **Broadcast workflow.** In professional NDI environments, video backgrounds are composited downstream by the vision mixer. Embedding MP4 playback in the scripture display duplicates work the broadcast chain already does, and reduces the video director's flexibility.
2. **Technical risk during live service.** A bad codec pack or driver-blacklisted GPU on the projection laptop can crash the app mid-service. Deferring video keeps the failure surface small.
3. **ROI.** Static backgrounds with optional gradient overlays cover the vast majority of real-world use. The remaining cases are already handled downstream.

### Decision 3 — Per-Channel Theme Scope: Display Window Only

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Display-window only (operator panel isolated via `theme.active_theme`)** | Channel themes affect the congregation `DisplayWindow` only. Operator panel chrome is driven by the existing `theme.active_theme` setting (default `"dark_gold"`). | **92/100** | **Selected** |
| Channel themes affect the operator panel's preview area for that channel | Mixed surfaces. UX becomes confusing when the preview tab matches a colorful theme but the panel chrome doesn't. | 50/100 | Rejected |
| Channel themes affect the entire operator panel | Impossible to operate in a dimly-lit booth if a channel theme is bright/colorful. | 12/100 | Rejected |

**Why display-only was selected:**

1. **The operator panel is a control surface, not a broadcast canvas.** It must stay dark, high-contrast, and ergonomically consistent across multi-hour services.
2. **Industry standard.** OBS, vMix, ProPresenter, and Resolve all strictly separate their control UI theme from output rendering.
3. **`generate_stylesheet()` is high-risk to swap dynamically.** The QSS generator at `src/utils/theme.py:90` controls spacing, border radii, scrollbar widths, and panel margins across the entire operator panel. Swapping per-channel would cause layout shifts during live operation.

**Implementation consequence:** The existing `theme.active_theme` setting (at `src/utils/settings.py:111-113`, currently unused — `main.py` hardcodes `DEFAULT_THEME`) is wired into startup to control operator panel chrome. `ChannelSettings.theme_id` controls only the per-channel `DisplayWidget` paint output (band color, background, logo, fonts, accent color).

### Decision 4 — Theme File Format: `.json` with `schema_version`

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **`.json` with `"schema_version": "2.0"` field** | Flat JSON. Human-editable, git-diff-friendly, universally toolable. No migration burden on the 3 existing built-ins. | **88/100** | **Selected** |
| Rename `.json` → `.vftheme` (still flat JSON) | Empty abstraction. No assets to bundle yet. Migration burden with zero benefit. | 30/100 | Rejected for v1.3.0 |
| `.vftheme` as a true zip archive (JSON manifest + `/assets/`) | Right answer when font and image bundling lands. Wrong answer for v1.3.0 because there is nothing to bundle yet. | 70/100 | Deferred to v1.3.x |

**Why JSON with schema version was selected:**

1. **Schema versioning** at the top of every theme file (`Theme.schema_version`) covers forward compatibility without changing the file format.
2. **Tooling.** Existing themes in `src/utils/themes/*.json` continue to load. `git diff` on a JSON file is meaningful; on a zip archive it is not.
3. **Future path is clear.** When custom font bundling and image asset packaging land in a v1.3.x release, `.vftheme` becomes a true `.zip` archive (mirroring `.obs-theme`, `.proBundle`, `.xptheme`). Reserve the extension for when there is something to bundle.

### Decision 5 — Custom Font Loading Strategy

`QFontDatabase.addApplicationFont()` registers fonts into Qt's global database. Loading the same `.ttf` repeatedly leaks GDI handles on Windows and produces duplicate enumeration entries.

**Decision:** Custom fonts are loaded **once at app startup** by a scan in `theme.py`. Themes reference fonts by `family` name only. Subsequent calls (e.g., after Font Import in the designer) register only newly-added files.

**Implementation:**

- New `src/utils/themes/fonts/` directory holds bundled `.ttf`/`.otf` files.
- `ThemeManager._load_application_fonts()` scans the directory. Tracks loaded files by **file path** (not a boolean flag) in `self._loaded_font_paths: set[str]`. Only registers files not yet in the tracking set. This allows the designer's Font Import to call the method again and pick up newly-copied files without re-registering existing ones.
- Theme JSON `fonts.family` fields are validated against `self.application_fonts`; unknown families fall back to the system default with a `WARNING`.

### Decision 6 — Theme Asset Path Portability

`background_image` and `logo_path` fields in theme JSON must be portable across machines.

**Decision:** All asset paths in theme JSON are **relative to the theme JSON file's parent directory**. Absolute paths are rejected at load time with a `WARNING` and treated as `null`. A `src/utils/themes/assets/` subdirectory convention is introduced for shared assets.

**Example:** A theme at `src/utils/themes/midnight_blue.json` with `"background_image": "assets/starfield.png"` resolves to `src/utils/themes/assets/starfield.png`.

## 3. Target Architecture

### 3.1 Current State (post v1.2.0)

```text
ThemeManager (single global instance)
  +-- _current = Theme         <-- shared across operator panel + both channels
  +-- set_theme(id) -> applies QSS to QApplication

ChannelManager
  +-- main = DisplayChannel(controller=DisplayController)
  |     +-- apply_settings({"theme_id": ...})  <-- calls ThemeManager.set_theme() globally
  +-- alt  = DisplayChannel(controller=DisplayController)
        +-- apply_settings({"theme_id": ...})  <-- overwrites Main's theme

DisplayWindow(QMainWindow).paintEvent (lower-third)
  +-- reads constants.LOWER_THIRD_BACKGROUND_ALPHA  <-- hardcoded
  +-- reads constants.LOWER_THIRD_HEIGHT_RATIO       <-- hardcoded
  +-- fills band with QColor(10, 10, 10, alpha)      <-- hardcoded
        ^ JSON declares lower_third.background_color but paintEvent ignores it
```

**Coupling problems this stage resolves:**

- `ThemeManager._current` is a single global -> Main's theme leaks into Alt and vice versa.
- `DisplayChannel.apply_settings()` calls global `set_theme()` -> unsolvable race when both channels persist different `theme_id`.
- Lower-third paint reads constants instead of theme JSON -> declared properties (`background_color`, `accent_color`, `separator_color`) are dead.
- `theme_id="default"` is the channel default but no theme has `id="default"` -> silent fallback + log spam on startup.
- Logo placeholder (`_lt_logo` `QFrame`) cannot accept a `QPixmap` -> roadmap blocker since v1.1.0.
- Guard scripts (`scripts/pre_commit_checks.py`, `scripts/verify_critical_fixes.py`) reference the old `2. Source Code/` path which no longer exists after the `src/` restructure.

### 3.2 Target State (post v1.3.0)

```text
Application startup
  +-- ThemeManager._load_application_fonts()  <-- one-time scan of themes/fonts/
  +-- ThemeManager._load_builtins()           <-- scans themes/*.json
       +-> each Theme has schema_version, validated against KNOWN_SCHEMA_VERSIONS

OperatorPanelTheme
  +-- theme.active_theme = "dark_gold"  <-- drives QApplication.setStyleSheet()
       ^ Stable. Never changed by per-channel theme application.

ChannelManager
  +-- main.theme = ThemeManager.get_theme(main_settings.theme_id)
  |     +-- DisplayChannel.set_theme(theme) -> forwards to DisplayWidget
  +-- alt.theme = ThemeManager.get_theme(alt_settings.theme_id)
        +-- DisplayChannel.set_theme(theme) -> forwards to DisplayWidget

DisplayWindow(QMainWindow) per channel
  +-- centralWidget() = DisplayWidget(QWidget)  <-- extracted rendering core
       +-- self._theme: Theme  <-- held instance, not global
            +-- paintEvent reads self._theme.lower_third.background_color
            +-- paintEvent reads self._theme.lower_third.background_alpha
            +-- _build_lower_third_page reads self._theme.lower_third.logo_path
            |     +-- QSvgWidget for SVG, QLabel(QPixmap) for raster
            +-- _render_fullscreen reads self._theme.fullscreen.background_image

ThemeDesignerPanel  (new -- opens from Settings -> "Open Theme Designer")
  +-- ThemesListPanel (left)         <-- saved themes + duplicate/delete
  +-- PreviewSurface (center)        <-- embedded DisplayWidget directly
  +-- PropertyEditor (right)         <-- color pickers, font selectors, sliders
        +-> live edits -> Theme.update() -> preview.refresh() -> real-time visual feedback
```

### 3.3 Per-Channel Theme Application Contract

```python
class DisplayWidget(QWidget):
    """Extracted rendering core. Contains all paint logic, font fitting,
    lower-third layout, fullscreen layout, and logo infrastructure.
    Can be embedded in DisplayWindow OR directly in ThemeDesignerPanel."""

    def set_theme(self, theme: Theme) -> None:
        """Apply a theme to this widget only. Channel-scoped.
        Does NOT call QApplication.setStyleSheet().
        Repaints the active mode (fullscreen or lower-third) once."""

class DisplayWindow(QMainWindow):
    """Thin shell. Window flags, screen targeting, headless mode.
    Delegates all rendering to self._display_widget (DisplayWidget)."""

    def set_theme(self, theme: Theme) -> None:
        """Forward to self._display_widget.set_theme()."""

class DisplayChannel(QObject):
    @property
    def theme(self) -> Theme: ...

    def set_theme(self, theme: Theme) -> None:
        """Forward to display_window.set_theme(); no-op when window is None."""

class ThemeManager(QObject):
    """Theme registry. App owns one shared instance.
    Each channel reads themes from this registry but holds its own
    current Theme reference on DisplayChannel.theme."""

    application_fonts: set[str]       # registered .ttf/.otf family names
    _loaded_font_paths: set[str]      # tracks which files have been registered
    set_app_theme(theme_id, app)      # operator panel chrome only
    get_theme(theme_id) -> Theme      # returns Theme or None (no fallback warnings)
```

## 4. Theme Schema v2 Specification

### 4.1 Top-Level Fields

```json
{
  "schema_version": "2.0",
  "name": "Dark Gold",
  "id": "dark_gold",
  "description": "Default dark theme with gold accents",
  "version": "1.0",
  "author": "VerseFlow Built-in",
  "colors": { ... },
  "fonts": { ... },
  "spacing": { ... },
  "animation": { ... },
  "lower_third": { ... },
  "fullscreen": { ... }
}
```

### 4.2 New / Extended Sections

`fullscreen` (new):

```json
"fullscreen": {
  "background_color": "#0a0a14",
  "background_image": null,
  "background_image_fit": "cover",
  "background_image_opacity": 1.0,
  "ref_color": "#c8a03c",
  "verse_color": "#e8e2d8",
  "ref_font_family": "Segoe UI",
  "ref_font_weight": "Black",
  "verse_font_family": "Segoe UI",
  "verse_font_weight": "Normal"
}
```

`lower_third` (extended -- fields already declared in `dark_gold.json` are now actually read by `paintEvent`):

```json
"lower_third": {
  "logo_path": null,
  "logo_format": "auto",
  "show_logo_placeholder": true,
  "logo_width_ratio": 0.10,
  "logo_max_height_ratio": 0.80,
  "show_separator": true,
  "separator_color": "#c8a03c",
  "separator_width": 1,
  "background_color": "#0a0a0a",
  "background_image": null,
  "background_image_fit": "cover",
  "background_alpha": 0.72,
  "height_ratio": 0.30,
  "accent_color": "#c8a03c",
  "ref_color": "#c8a03c",
  "verse_color": "#e8e2d8",
  "ref_font_family": "Segoe UI",
  "verse_font_family": "Segoe UI",
  "church_name_color": "#c8a03c",
  "transition": {
    "type": "fade",
    "duration_ms": 200,
    "easing": "OutCubic"
  }
}
```

`background_image_fit` accepts `"cover"`, `"contain"`, `"stretch"`, `"tile"`. All image paths are **relative to the theme JSON file's parent directory**. Absolute paths are rejected at load time. `transition.type` accepts `"none"` and `"fade"` in v1.3.0.

### 4.3 Schema Migration

`Theme.__init__` validates `schema_version` against `KNOWN_SCHEMA_VERSIONS = {"1.0", "2.0"}`. Schema 1.0 themes (the existing 3 built-ins) are auto-upgraded in memory at load time:

- Missing `fullscreen` section -> populated from `colors.bg_secondary`, `colors.text_primary`, `fonts.*`.
- Missing `schema_version` -> treated as `"1.0"`.
- Missing `lower_third` keys -> filled from `LOWER_THIRD_*` constants in `src/utils/constants.py`.

The on-disk files remain `schema_version: "1.0"` until the user re-saves them in the designer -- at which point they are written back as 2.0.
