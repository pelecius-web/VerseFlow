# VerseFlow v1.1.0 Advanced Display Modes — Sub-Roadmap

> **Stage:** v1.1.0  
> **Roadmap Category:** Category 2 — Advanced Display Modes  
> **Primary Goal:** Add professional lower-third rendering and dual-output channel architecture without regressing the current v1.0 single-display workflow.  
> **Implementation Style:** Gradual, guarded, regression-first.  
> **Status:** ✅ Complete (May 4, 2026)  

---

## 1. Executive Summary

v1.1.0 upgrades VerseFlow from a single-output Bible display application into a professional multi-output presentation system.

The stage introduces:

- **ChannelManager architecture** for Main/Alt outputs.
- **DisplayChannel abstraction** for independent output state.
- **Lower-third rendering mode** with explicit logo placeholder support.
- **Dual output channels** with independent live/clear/current/mode state.
- **Push Main / Push Alt / Push All** workflows.
- **Tabbed Main/Alt preview** in the operator panel.
- **Per-channel theme and mode foundation** for later v1.3.0 theme designer.

The critical implementation rule is:

> First preserve existing v1.0 behavior behind a new channel abstraction, then add new display modes and additional channels.

This avoids mixing feature work with fragile architectural migration.

---

## 1.1 Architectural Decision Record

### Decision

Use **Option A — Adapter-First Channel Architecture** for v1.1.0.

Do not replace `DisplayController` immediately. Instead, introduce `ChannelManager` and `DisplayChannel` as a wrapper layer around the existing `DisplayController`, treating the existing controller as the initial **Main** channel implementation.

### Options Considered

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Option A — Adapter-First Channel Architecture** | Preserve `DisplayController`, wrap it as Main, introduce ChannelManager gradually, then migrate consumers one section at a time. | **84/100** | **Selected** |
| **Option B — Full Channel Replacement Now** | Replace `DisplayController` immediately with a complete Main/Alt channel system and migrate all consumers at once. | 60/100 | Rejected for v1.1.0 |

### Why Option A Was Selected

- Lowest regression risk for the current operator workflow.
- Preserves existing `DisplayController` signals used by preview, queue, playlist, navigator, history, and hotkeys.
- Fits the current three-column `HomePanel` layout without forcing a disruptive UI rewrite.
- Allows lower-third rendering, dual output, NDI/API control, and per-channel themes to be built on a stable public output API.
- Avoids mixing a large architectural rewrite with new display-mode feature work.

### Long-Term Architecture Direction

The adapter-first approach is a stepping stone, not the final destination.

```text
v1.1.0 start:
MainWindow
  -> DisplayController
  -> ChannelManager
       -> main = DisplayChannelAdapter(existing DisplayController)

v1.1.0 later:
ChannelManager
  -> main = DisplayChannelAdapter(existing DisplayController)
  -> alt = DisplayChannel

future:
ChannelManager
  -> main = DisplayChannel
  -> alt = DisplayChannel
```

### Architectural Rules Going Forward

- `Main` remains the default output target for all existing workflows.
- New output features should call `ChannelManager`, not reach into UI widgets.
- Existing `display.*` consumers must be migrated gradually, not all at once.
- `DisplayController` must not keep growing indefinitely; later stages should extract responsibilities into channel, renderer, overlay, and window-management services.
- UI widgets should emit operator intent; output business logic should live in `ChannelManager` or channel-related services.
- Future v1.2.0 API/NDI and v1.3.0 theme features should integrate through `ChannelManager`.

## 3. Target Architecture

### 3.1 Current UI Discovery

```text
HomePanel
  ├── Left sidebar, fixed 280px
  │     ├── Home / Settings buttons
  │     ├── version badge
  │     └── DraftEditor
  ├── Center column, fixed 320px
  │     ├── TranslationMenu
  │     ├── SearchLineEdit
  │     ├── ModeToggle
  │     ├── verse navigation Up / Down
  │     ├── Display Controls: Show / Hide
  │     └── PlaylistPanel
  └── Right content, expanding
        ├── DISPLAY PREVIEW, fixed 300px
        ├── result stack: VerseNavigator / KeywordResults
        └── bottom panels: QueuePanel + LiveHistory
```

The current UI is already production-oriented and should not be disrupted. The architectural change must therefore fit into the existing three-column structure.

Important existing coupling:

- `MainWindow` creates one `DisplayController` and passes it into `HomePanel`.
- `HomePanel` passes the same display object to `DisplayPreview`, `VerseNavigator`, `KeywordResults`, `QueuePanel`, `PlaylistPanel`, and `VerseLiveHistoryPanel`.
- `DisplayPreview` listens directly to `display.verse_changed` and `display.translations_changed`.
- Queue, playlist, and navigator live-state sync depends on `display.verse_changed`.
- Existing hotkeys call `HomePanel._on_show()` and `HomePanel._on_hide()`, which currently target the single display.

Because of this, **ChannelManager must first wrap the existing `DisplayController` instead of replacing it outright**.

### 3.2 Target v1.1.0 Model

The long-term model remains:

```text
HomePanel
  -> ChannelManager
       -> Main DisplayChannel
            -> DisplayWindow
            -> DisplayPreview
            -> current verse
            -> visibility state
            -> display mode
            -> theme id
       -> Alt DisplayChannel
            -> DisplayWindow
            -> DisplayPreview
            -> current verse
            -> visibility state
            -> display mode
            -> theme id
```

However, the first implementation model should be safer:

```text
MainWindow
  -> existing DisplayController
       -> treated as Main channel implementation for Phase 1
  -> ChannelManager
       -> main = existing DisplayController
       -> alt = added later

HomePanel
  -> continues receiving display-compatible object at first
  -> gradually receives ChannelManager for new channel-specific UI
```

This means Phase 1 is a **wrapper/adapter phase**, not a full replacement phase.

### 3.3 Migration Principle

Do not immediately rewrite all `display.*` consumers.

Migrate in this order:

1. Create `ChannelManager` wrapper around existing `DisplayController`.
2. Preserve all existing `DisplayController` signals and methods.
3. Keep Main as the default target.
4. Move center-column Show/Hide controls to call ChannelManager.
5. Move preview to channel-aware preview.
6. Move navigator, keyword results, queue, playlist, and history one at a time.
7. Add Alt only after Main remains stable through the wrapper.

### 3.4 Core New Components

#### DisplayChannel

Represents one output destination.

Responsibilities:

- Own or reference one display window.
- Track current live verse.
- Track whether output is visible/live/clear.
- Track display mode: fullscreen or lower-third.
- Track theme id.
- Push verse to that channel only.
- Clear that channel only.
- Emit state-change signals.

Suggested public API:

```python
class DisplayChannel(QObject):
    verse_changed = pyqtSignal(dict)
    cleared = pyqtSignal()
    visibility_changed = pyqtSignal(bool)
    mode_changed = pyqtSignal(str)
    state_changed = pyqtSignal(str, dict)

    def push_verse(self, verse: dict): ...
    def clear(self): ...
    def show(self): ...
    def hide(self): ...
    def set_display_mode(self, mode: str): ...
    def set_theme(self, theme_id: str): ...
    def get_state(self) -> dict: ...
```

Phase 1 may implement `DisplayChannel` as a thin adapter over the existing `DisplayController` instead of moving all display logic immediately.

Suggested adapter shape:

```python
class DisplayChannel(QObject):
    def __init__(self, name, controller):
        self.name = name
        self.controller = controller

    @property
    def current(self):
        return self.controller.current

    def push_verse(self, verse):
        self.controller.push_verse(verse)

    def clear(self):
        self.controller.push_verse({})
```

The adapter must forward or expose:

- `verse_changed`
- `translations_changed`
- `display_opened`
- `display_closed`
- `fullscreen_toggled`

#### ChannelManager

Coordinates all output channels.

Responsibilities:

- Create and register Main and Alt channels.
- Route push/clear actions to the correct channel.
- Preserve compatibility with existing single-display workflows.
- Provide Push All and Clear All.
- Expose state for preview tabs and status badges.

Suggested public API:

```python
class ChannelManager(QObject):
    channel_changed = pyqtSignal(str, dict)
    all_channels_changed = pyqtSignal(dict)

    def get_channel(self, name: str): ...
    def push_to_channel(self, name: str, verse: dict): ...
    def clear_channel(self, name: str): ...
    def push_to_all(self, verse: dict): ...
    def clear_all(self): ...

    # Temporary compatibility layer for existing v1.0 behavior
    def push_verse(self, verse: dict): ...
    def clear(self): ...
    def show(self): ...
    def hide(self): ...
    @property
    def current(self): ...
```

In Phase 1, compatibility methods must delegate to Main:

```python
def push_verse(self, verse):
    return self.push_to_channel("main", verse)

def clear(self):
    return self.clear_channel("main")

@property
def current(self):
    return self.get_channel("main").current
```

This preserves old workflows while new channel-aware UI is introduced.

# Phase 1 — ChannelManager Wrapper Foundation

## Objective

Introduce the channel architecture as a wrapper around the existing `DisplayController`, while preserving current single-display behavior exactly.

No lower-third rendering, Alt output, or visible UI redesign should be added in this phase.

This is deliberately an adapter phase. The existing `DisplayController` already behaves like a single channel because it owns `current`, `display_window`, `secondary_translations`, `push_verse()`, `clear()`, `verse_changed`, and `translations_changed`. Therefore, the first safe step is to treat the existing controller as the Main channel implementation.

## Deliverables

- New `display_channel.py`.
- New `channel_manager.py`.
- `DisplayChannel` can wrap an existing `DisplayController`.
- `ChannelManager` registers `main` using the existing `DisplayController`.
- Existing `HomePanel` can continue using `self.display` during the transition.
- New code can use `self.channel_manager` for channel-aware operations.
- Compatibility methods preserve old `display.push_verse`, `display.clear`, `display.current`, `verse_changed`, and `translations_changed` expectations.

## Files Likely Modified

- `main.py`
- `home_panel.py`
- `display_core.py` only if unavoidable
- `pre_commit_checks.py`
- `verify_critical_fixes.py`
- New: `display_channel.py`
- New: `channel_manager.py`

## Implementation Steps

1. Create `DisplayChannel` as a thin adapter over an existing display-like object.
2. Create `ChannelManager` with only `main` registered.
3. In `main.py`, keep creating the existing `DisplayController`.
4. Create `ChannelManager(main_controller=self.display)` after `DisplayController`.
5. Pass the existing `DisplayController` to `HomePanel` as before.
6. Optionally pass `ChannelManager` to `HomePanel` as a second dependency for future channel UI.
7. Do not rewrite `VerseNavigator`, `KeywordResults`, `QueuePanel`, `PlaylistPanel`, `DisplayPreview`, or `VerseLiveHistoryPanel` yet.
8. Add compatibility methods on `ChannelManager` that delegate to Main.
9. Verify current app behavior before adding any lower-third or Alt-channel feature.

## Current UI-Specific Constraints

- The center column is fixed at 320px and already contains `DISPLAY CONTROLS`; future channel target/mode controls should be added here, not in the navigator area.
- The right preview box is fixed at 300px and currently contains a single `DisplayPreview(self.display)`; channel preview tabs should be planned around this height constraint.
- `display.verse_changed` currently drives queue live state, playlist live state, navigator state sync, and preview updates; this signal must remain available and must still represent Main during migration.
- Existing hotkeys call `HomePanel._on_show()` and `HomePanel._on_hide()`; they must continue to operate on Main only.

## Regression Hazards

- `display.verse_changed` signal no longer reaches existing sync logic.
- Queue/playlist live-state sync breaks.
- Navigator state sync breaks.
- Hotkeys push to wrong object.
- History restore no longer updates display.
- Existing display preview stops updating.
- `DisplayPreview` stops receiving `translations_changed`.
- `self.display.current` compatibility breaks in queue/playlist sync code.

Add to `pre_commit_checks.py`:

- `channel_manager.py` exists.
- `class DisplayChannel` exists.
- `class ChannelManager` exists.

## Verification Guards

### v1.1.2 Architectural Changes (May 7, 2026)

The following changes were made to the Phase 3 lower-third implementation after initial code completion:

1. **Square logo** — `_lt_logo` is constrained to `setFixedSize(inner_w, inner_w)` where `inner_w = container_width - horizontal_padding`. The logo was previously a tall rectangle (1:1.8 aspect) that stretched to fill the container. The square shape frees 50–80px of vertical space below.

2. **Church alias label** — New `_lt_church_label` (QLabel) sits below the square logo in the same `QVBoxLayout`. Text is sourced from `SettingsManager.get("general", "church_name", "")`. Font is fitted independently via `_fit_church_name_font()` (downward scan from 11pt). If empty or too long, the label is hidden with zero space. Users enter short abbreviations (e.g. "G A Zone") via Settings → General → Church Alias.

3. **Independent reference font** — `_fit_reference_font()` fits the reference label into a dedicated `LOWER_THIRD_REF_ROW_HEIGHT = 36px` row. The verse binary search now operates on `verse_avail_h = avail_h - 36 - 4 - SAFETY_BUFFER`, eliminating the feedback loop where `ref_font = 0.60 × verse_font` caused the ref to steal from the verse budget.

4. **Removed line-count heuristic** — `_fit_church_name_font` originally used `approx_lines = round(h / (size * 1.4))` which falsely rejected short abbreviations due to QTextDocument's internal documentMargin inflating measured heights. Now uses measured pixel height only (`h <= avail_h`).

5. **Container padding reduced** — Logo container horizontal padding changed from 12→6px, giving the church alias label more text width at small resolutions.
