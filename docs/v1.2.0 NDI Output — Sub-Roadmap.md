# VerseFlow v1.2.0 NDI Output — Sub-Roadmap

> **Stage:** v1.2.0 — ALL PHASES COMPLETE (May 13, 2026 — NDI output full feature: bridge, capture, alpha, visibility, persistence)  
> **Roadmap Category:** Category 3 — Integration & Output (NDI Only)  
> **Primary Goal:** Add professional NDI output for Main and Alt channels with alpha channel support, without touching any existing operator workflow, compatibility shim, or display controller.  
> **Status:** ✅ COMPLETE
> **Implementation Style:** Additive only. Zero regression to v1.1.0 behavior.

---

## 1. Executive Summary

v1.2.0 adds NDI (Network Device Interface) output to VerseFlow, enabling live video feeds of the congregation display to be received by OBS, vMix, Wirecast, and other broadcast tools over the local network.

The stage introduces:

- **`ndi_bridge.py`** — a minimal ctypes wrapper over the NDI C SDK. No PyPI dependency.
- **`NDISender`** — a pure output sink that subscribes to `ChannelManager.channel_changed` and captures frames from `DisplayWindow` via `QWidget.grab()` into an ARGB32 image.
- **Per-channel NDI sources** — `"VerseFlow Main"` and `"VerseFlow Alt"` as independent named NDI sources on the local network.
- **Alpha channel transparency** — lower-third mode transmits RGBA frames so OBS can composite the overlay cleanly over any video background.
- **Operator health UI** — a compact status widget showing sender state and FPS, placed below the existing preview panel.
- **Settings integration** — `ndi_enabled` and `ndi_source_name` per channel, extending the existing `ChannelSettings` pattern.

The critical implementation rule is:

> `NDISender` is a **read-only observer**. It connects to one signal and never writes to `DisplayController`, `ChannelManager`, `HomePanel`, or any operator widget.

The HTTP/WebSocket API and Bitfocus Companion module originally planned for this version have been **deferred indefinitely**. They are not needed for the AI-powered, minimal-supervision workflow that is VerseFlow's primary goal. NDI alone solves the professional broadcast output problem.

## 1.1 Architectural Decision Record

### Decision 1 — Frame Capture Method

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **`QWidget.grab()` → `convertToFormat(ARGB32)`** | Forces Qt to render the widget tree directly into a pixmap via `paintEvent()`. Bypasses backing store and OS-level surface concerns. Works for GPU-accelerated, stay-on-top, translucent, and hidden (headless) windows. | **85/100** | **Selected** |
| `QWidget.render()` into ARGB32 QImage | Off-screen render into alpha-capable surface. Non-blocking. Genuine transparency. **Rejected after testing:** empty for fullscreen windows with `WindowStaysOnTopHint` and `WA_TranslucentBackground` on Windows — reads from Qt's backing store, which these window types don't populate. | 28/100 | Rejected (tested — doesn't work on Windows) |
| `QScreen.grabWindow()` | Synchronous screenshot via `GetWindowDC()` + `BitBlt()`. **Rejected:** reads the GDI surface — empty for GPU-accelerated (Direct2D/DirectComposition) windows. Comment claiming it "reads from the DWM compositor" was factually incorrect. | 0/100 | Rejected |

**Why `grab()` was selected over the original `render()` plan:**

Both `render()` and `grabWindow()` were tested on Windows 10/11 and both failed to capture GPU-accelerated window content. `render()` reads from Qt's backing store which is empty for `WindowStaysOnTopHint` + `showFullScreen()` windows. `grabWindow()` reads the GDI surface which GPU content never touches. `QWidget.grab()` triggers a fresh paint cycle into a pixmap, making it the only viable option for the DisplayWindow's window flags and rendering pipeline.

**Alpha trade-off:** `grab()` does not preserve per-pixel alpha — the pixmap has the composited window background. Fullscreen mode is correct (fully opaque). Lower-third mode's transparent area shows the window background rather than true alpha. Phase 3 alpha validation will determine whether a mode-specific capture strategy is needed for lower-third transparency.

### Decision 2 — NDI SDK Integration Method

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **`ndi_bridge.py` ctypes wrapper** | ~280 lines of Python ctypes calling NDI C DLL directly (struct definitions, frame cache, error handling). Zero pip dependencies. Ships with app. | **90/100** | **Selected** |
| Third-party PyPI package (e.g., `ndi-python`) | Pre-built binding. Opaque versioning. Risk of unmaintained package. Pip dependency in production. | 55/100 | Rejected |

**Why ctypes was selected:**

VerseFlow needs only six NDI SDK functions. A thin ctypes wrapper is fully self-contained, has no external package requirement, and is not subject to breaking changes from an unmaintained PyPI package. The NDI runtime DLL ships alongside the application.

### Decision 3 — Capture Trigger

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **QTimer at 33ms (30fps)** | Continuous capture loop while live. Matches broadcast expectations. Lower-third appears smooth on stream even when content is static. | **88/100** | **Selected** |
| Signal-driven (capture on verse change only) | Simple. No timer. But produces a slideshow-style NDI feed — static frames with no intermediate updates, which looks wrong in broadcast tools. | 40/100 | Rejected |

**Supplemental note:** Fullscreen mode (static Bible text) could theoretically drop to 10fps without quality loss, but a consistent 30fps timer across all modes is simpler to implement and maintain. Mode-dependent fps logic adds complexity for negligible bandwidth savings on a local network.

**Phase 1 measurement requirement:** The 30fps assumption must be validated on real hardware before being committed. Phase 1 must log `grab()` duration in `DEBUG` mode on every frame tick. The target fps is controlled by a single `NDI_CAPTURE_FPS = 30` constant in `ndi_sender.py` so the interval can be adjusted without a code search. If `grab()` consistently exceeds `(1000 // NDI_CAPTURE_FPS) - 5` ms, a `WARNING` is logged and the constant should be reduced. If `grab()` exceeds the full interval (33ms at 30fps), Qt will queue the next slot naturally — frames drop rather than stack, so the event loop remains responsive.

### Decision 4 — API Scope

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **NDI output only** | Delivers the broadcast output goal. No network-facing server. No security surface. Aligns with AI-supervised, minimal-operator workflow. | **95/100** | **Selected** |
| NDI + HTTP REST + WebSocket API | Full external control surface. Needed for multi-operator AV setups and Stream Deck integration. Not needed when AI is the primary operator. | 62/100 | Deferred |

## 3. Target Architecture

### 3.1 Current Output Layer (post v1.1.0)

```text
MainWindow
  → DisplayController (main channel implementation)
  → ChannelManager
       → main = DisplayChannel(adapter over DisplayController)
       → alt  = DisplayChannel(adapter over DisplayController)

HomePanel
  → self.main_display = ChannelDisplayFacade(channel_manager, "main")
       ← DisplayController signals forwarded through facade
  → self.channel_manager = ChannelManager
       ← used for new channel-aware UI (mode controls, preview tabs)
```

**Key coupling facts:**

- `ChannelDisplayFacade` (`channel_display_facade.py`) is `self.main_display` in `main.py`. All legacy operator widgets (navigator, queue, playlist, history, preview) hold a reference to this facade. It must not be modified.
- `ChannelManager.channel_changed` fires on every channel state transition with a normalized `state_dict`. This is the only signal `NDISender` needs to connect to.
- `DisplayWindow` is created lazily when the first verse goes live. `NDISender` must tolerate `display_window` being `None` between verses.

### 3.2 v1.2.0 Target: NDI as a Pure Output Sink

```text
MainWindow
  → DisplayController
  → ChannelManager
       → main = DisplayChannel
            → DisplayWindow  ←── NDISender["main"] captures frames here
       → alt  = DisplayChannel
            → DisplayWindow  ←── NDISender["alt"] captures frames here
  → NDIManager
       → NDISender["main"]  ← connects to channel_manager.channel_changed
       → NDISender["alt"]   ← connects to channel_manager.channel_changed
```

`NDIManager` is a lightweight coordinator (similar to `ChannelManager`) that:
- Creates `NDISender` instances keyed by channel name.
- Passes `channel_manager` to each sender.
- Handles app-level NDI SDK lifecycle (`NDIlib_initialize` / `NDIlib_destroy`).

`NDISender` is a `QObject` that:
- Subscribes to `channel_manager.channel_changed`.
- On `is_live` → true: acquires `DisplayWindow` reference, starts `QTimer` at 33ms.
- On timer tick: calls `grab()` into ARGB32, sends frame via `ndi_bridge`.
- On `is_live` → false: stops timer, destroys NDI sender handle.
- Emits its own signals: `frame_sent`, `sender_error`, `sender_status_changed`.
- Never reads `DisplayController`, `HomePanel`, or any operator widget directly.

### 3.3 NDI Frame Capture Contract

```python
class NDISender(QObject):
    # Signals — read-only output. Health widget connects here.
    frame_sent      = pyqtSignal(str, int)   # (channel_name, fps)
    sender_error    = pyqtSignal(str, str)   # (channel_name, error_message)
    sender_status_changed = pyqtSignal(str, dict)  # (channel_name, status_dict)

    def __init__(self, channel_name: str, channel_manager, ndi_bridge, parent=None): ...
    def start(self): ...   # called when channel goes live
    def stop(self): ...    # called when channel clears; sends black frame before teardown
    def _on_timer(self): ...   # 33ms capture loop
    def _capture_frame(self) -> QImage | None: ...
        # Uses QWidget.grab() → pixmap.toImage().convertToFormat(ARGB32)
        # Tolerates headless windows (single-monitor) via getattr(..., '_headless')
    def _send_frame(self, image: QImage): ...
```

| Channel | Default NDI Source Name | Configurable |
|---------|------------------------|--------------|
| Main | `"VerseFlow Main"` | Yes — via settings |
| Alt | `"VerseFlow Alt"` | Yes — via settings |

## 4. NDI Bridge Specification (`ndi_bridge.py`)

### 4.1 Required SDK Functions

| Function | Purpose |
|----------|---------|
| `NDIlib_initialize()` | Load and initialize the NDI runtime. Call once at app startup. |
| `NDIlib_send_create()` | Create a named NDI sender. One per live channel. |
| `NDIlib_send_send_video_async_v2()` | Push an RGBA or BGRA frame asynchronously. Does not block Qt event loop. |
| `NDIlib_send_destroy()` | Destroy sender handle when channel clears or app exits. |
| `NDIlib_destroy()` | Unload SDK. Call once at app shutdown. |
| `NDIlib_send_get_tally()` | Optional: query whether the source is on-air at a downstream switcher. |

### 4.2 Frame Format

| Property | Fullscreen Mode | Lower-Third Mode |
|----------|----------------|-----------------|
| Pixel format | `BGRA` (NDI_FourCC_video_type_BGRA) | `BGRA` with alpha |
| Frame rate | 30fps (33ms timer) | 30fps (33ms timer) |
| Resolution | Match `DisplayWindow` geometry | Match `DisplayWindow` geometry |
| Alpha handling | All pixels fully opaque | Transparent above band, opaque within band |

### 4.3 Sender Lifecycle

```text
App startup:
  NDIlib_initialize() → stored in ndi_bridge singleton

Channel goes live (is_live: false → true):
  NDIlib_send_create(source_name) → store handle
  QTimer.start(33ms)

Timer tick (channel is live):
  pixmap = display_window.grab()
  image = pixmap.toImage().convertToFormat(QImage.Format.Format_ARGB32)
  NDIlib_send_send_video_async_v2(handle, frame_from_image)
  self._frame_buf = image  ← retain for async SDK
  emit frame_sent(channel_name, current_fps)

Channel clears (is_live: true → false):
  QTimer.stop()
  NDIlib_send_destroy(handle)
  handle = None
  emit sender_status_changed(channel_name, {"active": False})

Display mode changes while live:
  No action needed — grab() captures whatever DisplayWindow paints
  on the next timer tick. Mode changes are reflected naturally
  without destroying or recreating the NDI sender handle.

App shutdown:
  stop() all active senders
  NDIlib_destroy()
```

### 4.4 NDI Bridge Interface Contract

The `ndi_bridge.py` module exposes a single frame-send function to `NDISender`:

```python
ndi_bridge.send_frame(handle, qimage: QImage) → None
```

| | |
|---|---|
| **Precondition** | `qimage` may be in any `QImage.Format`. The caller (`NDISender`) never performs format conversion. |
| **Postcondition** | Frame is transmitted to the NDI network in the correct byte order for the SDK. |
| **Responsibility** | All pixel format conversion (ARGB32 → RGBA8888 → BGRA channel swap) is internal to `ndi_bridge.py`. |

`NDISender` must never call `convertToFormat()`, handle raw bytes, or reference any `QImage.Format` constant related to conversion. If `NDISender` ever performs format conversion, that is an architecture violation and a pre-commit guard failure.

## 5. Alpha Channel Specification

### 5.1 Why Alpha Matters

In lower-third mode, `DisplayWindow` occupies the full screen but only the bottom band is painted. The area above the band is transparent (set via `WA_TranslucentBackground`). For OBS to receive this as a usable overlay:

- The NDI frame must carry per-pixel alpha.
- The NDI SDK must transmit in RGBA/BGRA format (NDI 6.3 supports this).
- OBS must configure the source as an "NDI Source" overlay, not a background fill.

If alpha is lost anywhere in this chain, OBS receives a rectangle with a black or wallpaper-filled top section — unusable as a lower-third graphic.

### 5.2 Widget Capture Method (Production-Verified)

Both `render()` and `grabWindow()` were tested and **failed** on Windows GPU-accelerated windows. `render()` reads from Qt's backing store which is empty for `WindowStaysOnTopHint` + `showFullScreen()` windows. `grabWindow()` reads the GDI surface via `GetWindowDC()` + `BitBlt()` — GPU content never touches this surface.

`QWidget.grab()` triggers a fresh paint cycle into a pixmap, making it the only viable option.

**Alpha trade-off:** `grab()` composites the window background into the pixmap — it does not preserve per-pixel alpha. Fullscreen mode is correct (fully opaque). Lower-third mode's transparent area shows the window background rather than true alpha. Phase 3 alpha validation will determine whether a mode-specific capture strategy is needed for lower-third transparency.

Implementation in production (verified via Bug 58):

```python
def _capture_frame(self) -> QImage | None:
    channel = self._channel_manager.get_channel(self.channel_name)
    if channel is None:
        return None
    display_window = channel.display_window
    if display_window is None:
        return None
    # Headless windows (single-monitor NDI) are never shown —
    # skip visibility check for them.
    if not getattr(display_window, '_headless', False) and not display_window.isVisible():
        return None
    size = display_window.size()
    if size.isEmpty():
        return None
    pixmap = display_window.grab()
    if pixmap.isNull():
        return None
    image = pixmap.toImage().convertToFormat(QImage.Format.Format_ARGB32)
    return image
```

### 5.3 Conversion to NDI Frame

With `grab()`, the pixmap is already composited (fullscreen = fully opaque, lower-third = window background in the transparent region). The QImage is converted to ARGB32 for the SDK:

```python
image = pixmap.toImage().convertToFormat(QImage.Format.Format_ARGB32)
```

The `ndi_bridge.py` wrapper handles the ARGB→BGRA channel swap internally so `NDISender` always works with `QImage` objects only.

### 5.4 Validation Requirements (Phase 3)

Alpha is only correct if all of the following are confirmed:

- [ ] Lower-third band area is fully opaque in the NDI frame.
- [ ] Area above the band is fully transparent (alpha = 0) in the NDI frame.
- [ ] OBS NDI source composites cleanly over a test video clip with no black halo.
- [ ] After `clear()`, NDI sender is destroyed and OBS source shows "no signal" (not a frozen frame).
- [ ] After mode switch from lower-third → fullscreen while live, new frames are fully opaque.
- [ ] After mode switch from fullscreen → lower-third while live, alpha is restored correctly.
- [ ] On monitor disconnect while live, NDI sender does not crash — degrades to "no signal".

## Implementation Steps

1. Write `ndi_bridge.py` with ctypes bindings for the six required SDK functions.
2. Add graceful fallback if NDI DLL is not found — log warning, disable NDI silently, do not crash.
3. Write `NDISender.__init__` to accept `channel_name`, `channel_manager`, and `ndi_bridge`.
4. Connect `channel_manager.channel_changed` to `NDISender._on_channel_state(name, state)`.
5. In `_on_channel_state`: if `name != self.channel_name`, return immediately.
6. If `state["is_live"]` transitions false → true: call `self.start()`.
7. If `state["is_live"]` transitions true → false: call `self.stop()`.
8. `start()`: acquire `DisplayWindow` reference via `channel_manager.get_channel(name).display_window` (forwarding property on `DisplayChannel`). Start `QTimer` at 33ms.
9. Add `NDI_CAPTURE_FPS = 30` constant at module top. Use `QTimer.start(1000 // NDI_CAPTURE_FPS)`.
10. `_on_timer()`: record `t0 = time.perf_counter()`. Call `_capture_frame()`. If result is not None, call `_send_frame(image)` first (submit new frame before dropping old buffer), then `self._frame_buf = image` (now safe — SDK is done with previous buffer). Log grab duration at `DEBUG`. Log `WARNING` if duration exceeds `(1000 // NDI_CAPTURE_FPS) - 5` ms. Emit `frame_sent`.
11. `_capture_frame()`: use `display_window.grab()` to get a `QPixmap`, convert to ARGB32 via `pixmap.toImage().convertToFormat(QImage.Format.Format_ARGB32)`. Return `None` if window is `None`, not visible (non-headless), or zero-sized.
12. `_send_frame(image)`: pass `image` to `ndi_bridge.send_frame(handle, image)`. All format conversion is internal to `ndi_bridge`. `NDISender` does not call `convertToFormat()`.
13. `stop()`: stop timer, call `ndi_bridge.destroy_sender()`, set handle and `self._frame_buf` to None, emit `sender_status_changed`.
14. Write `NDIManager` to call `NDIlib_initialize()` at startup and `NDIlib_destroy()` at shutdown.
15. In `main.py`: create `NDIManager(channel_manager)` after `ChannelManager` is initialized.

## Regression Hazards

- `channel_manager.channel_changed` already has existing consumers. Adding `NDISender` as another consumer must not interfere with existing connections.
- `DisplayWindow` reference via `display_window` may be `None` between verses. `_capture_frame()` must handle `None` gracefully and return `None`.
- `NDIlib_initialize()` must not block the Qt event loop — it is synchronous but fast (<5ms typical).
- NDI DLL not found must degrade silently — not crash the app.
- **Frame lifetime (critical):** `NDIlib_send_send_video_async_v2()` returns before the SDK finishes reading the frame buffer. Python's GC will free a `QImage` immediately after the local variable goes out of scope. Required pattern: `self._frame_buf = image` before calling `_send_frame()`. This retains the buffer alive until the next frame replaces it. Clearing `self._frame_buf = None` in `stop()` releases it after the sender is destroyed. Do not use a bare local variable as the frame source for async send.

## 10. Operator Safety Guidelines

### Output Safety

- NDI output is read-only. Operator cannot accidentally push a verse via NDI.
- NDI sources are local-network-only (NDI does not route over the internet by default).
- Disabling NDI for a channel via settings is immediate — no restart required.
- NDI sender destruction on `clear()` ensures no stale frames are visible in OBS.

### Failure Modes

- NDI DLL not found: silent degradation. App runs fully without NDI. Warning in log.
- NDI SDK initialization failure: same — silent degradation, warning logged.
- Frame capture failure (window not visible): no crash, sender emits no-signal status.
- Sender creation failure (name conflict, SDK error): logged, status widget shows red dot, no crash.
- App close with active senders: SDK teardown called in `NDIManager.__del__` or `closeEvent`.

### Performance

- `QTimer` at 33ms on the Qt main thread: safe because `grab()` is fast (<2ms typical for a lower-third widget).
- If `grab()` takes longer than 33ms on slow hardware, frames will drop naturally — the timer fires after the previous slot completes.
- No separate threads required. Qt event loop remains responsive.
