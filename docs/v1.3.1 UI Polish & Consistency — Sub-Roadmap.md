# VerseFlow v1.3.1 UI Polish & Consistency — Sub-Roadmap

> **Stage:** v1.3.1 — Planned (May 21, 2026)
> **Roadmap Category:** Category 4a — Customization & Theming (Polish)
> **Primary Goal:** Bring the operator panel chrome (settings, home, theme designer) up to the same professional bar that the v1.3.0 Theme Designer established, by completing the property-based QSS migration, fixing the typographic hierarchy, and standardizing iconography across all panels.
> **Status:** Phase 0 ✅, Phase 1 ✅, Phase 2 ✅, Phase 3 ✅, Phase 4 ✅ — v1.3.1 COMPLETE
> **Implementation Style:** Additive QSS rules + targeted refactors. Zero regression to v1.3.0 channel/theme behavior. Operator panel layouts may shift; congregation display rendering does not change.

---

## 1. Executive Summary

v1.3.1 completes the operator-panel chrome work that v1.3.0 started. The Theme Designer and Settings panel both received polish attention during v1.3.0, but the work was uneven: the designer ships with a coherent visual language while the rest of the operator UI still uses inline rgba values, mismatched typographic scales, and ad-hoc layouts inherited from the v0.7 MVP. This stage closes that gap.

The stage delivers five capabilities:

- **Token-based typography system** — Phase 0 introduces a `typography` section in every theme JSON. `generate_stylesheet()` emits typography selectors via a loop. Adding a new size is a JSON edit, not a code edit. Themes can override per-scale (e.g., high contrast may want larger sizes for accessibility).
- **Unified property-based QSS pattern** — Every operator panel uses `setProperty()` selectors instead of inline `setStyleSheet()` calls. Single source of truth in `theme.py:generate_stylesheet()`.
- **Typographic hierarchy** — Section titles dominate their content. The 9px uppercase letter-spaced label pattern is reserved for compact contexts (sidebars, inline metadata). Full-width sections get 13–14px normal-case headings. Both sizes are entries in the typography system, not hardcoded selectors.
- **Section iconography** — Every section header carries a 16px icon. Operators locate sections by glyph in under one second, not by reading.
- **Layout fragility fixes** — Rows that overflow on narrow windows reflow gracefully. Hover affordances are reserved for interactive surfaces.

The critical implementation rule is:

> The QSS generated by `theme.py:generate_stylesheet()` is the single source of truth for operator panel chrome. After v1.3.1, no operator-panel `.py` file shall contain `setStyleSheet()` with hardcoded rgba/hex values except for ephemeral debug widgets explicitly marked TODO.

Six things explicitly **deferred** to a later release:

- **Sidebar navigation pattern (replacing "Back to Home" button)** — Requires architectural changes to `MainWindow.stack` routing. Tracked separately for a v1.4.0 design exploration.
- **Animated section transitions** — Out of scope; no current request.
- **Dark/light mode auto-detection from OS** — Tracked as a v1.4.x feature.
- **Custom user-defined typography scales** — Theme designer scope, not chrome scope.
- **Accessibility audit and WCAG compliance pass** — Worthwhile but requires manual testing with assistive tech; tracked separately.
- **`icons.py` factory pattern refactor (e.g., enum-based registry)** — Internal cleanup with no user-visible benefit; not justified by current scope. (B3)

## 2. Architectural Decision Record

**Scoring rubric:** Scores reflect consistency impact (35%), regression risk (25%), maintainability gain (25%), schedule fit (15%).

### Decision 1 — QSS Migration Strategy: Per-Panel, Not Big-Bang

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Per-panel migration with shared QSS rule additions per phase** | Each phase migrates one panel and adds whatever QSS rules that panel needs. Rules accumulate in `generate_stylesheet()` incrementally. Risk is bounded to one panel at a time. | **92/100** | **Selected** |
| Big-bang migration of all panels in one phase | Faster on paper. In practice, 3 panels × ~25 inline calls × 4 themes = high regression surface. One bad cascade rule breaks every panel simultaneously. | 55/100 | Rejected |
| Migration via codemod / automated AST rewrite | Inline stylesheets are not mechanically convertible — many encode design decisions that need human review (e.g., the error label's red value). Codemod produces wrong results. | 25/100 | Rejected |

**Why per-panel was selected:**

1. **Bounded blast radius.** A typo in a property selector affects one panel until it's tested. After validation, the rule is reusable for the next panel.
2. **Iterative validation.** Each phase ships visible improvement. If schedule pressure forces a cut, the work already shipped is already useful.
3. **Cascade order discoverability.** Adding QSS rules incrementally surfaces cascade conflicts as they appear, not all at once.
4. **The settings panel migration (v1.3.0 last week) is the reference implementation.** Its QSS additions (cards, section headers, gold dots, separators, accent variants) are the foundation. Subsequent phases extend them.

**Long-term consequence:** After v1.3.1, the property-based pattern is the canonical way to style operator-panel widgets. Future panels (v1.4.0 transcript history, v2.0.0 sermon notes export) inherit the system without inventing their own.

### Decision 2 — Typographic Hierarchy: Two Scales, Not One

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Two-scale system: `compact` (9px uppercase letterspaced) for sidebars/inline metadata, `standard` (13-14px normal case) for full-width sections** | Captures the legitimate use of the existing pattern (compact contexts) while fixing its misuse in full-width contexts. | **88/100** | **Selected** |
| Replace the 9px uppercase pattern entirely with 13-14px normal case everywhere | Loses the visual identity of the home panel sidebar where the compact pattern reads correctly. Forces a more intrusive home panel rework. | 60/100 | Rejected |
| Keep current 9px uppercase pattern as the only system header style | Status quo. Inverted hierarchy in full-width settings panel persists. Does not solve the problem. | 30/100 | Rejected |

**Why two-scale was selected:**

1. **Context-appropriate sizing.** A 280px sidebar at 9px is dense and legible. A 600px+ settings panel at 9px is anemic.
2. **Zero loss of existing identity.** The home panel sidebar's visual character is preserved.
3. **Industry alignment.** Linear, Notion, macOS System Settings, ProPresenter all use compact labels in chrome and standard labels in content. Two scales is the norm.
4. **Encoded in QSS, not duplicated.** Two property selectors (`section-header-compact`, `section-header-standard`) plus a default. Both live in `generate_stylesheet()`.

**Implementation consequence:** v1.3.0's `QLabel[section-header="true"]` selector is renamed to `QLabel[section-header="compact"]` for backward compatibility. A new `QLabel[section-header="standard"]` selector is added. The settings panel migrates to the standard variant.

### Decision 3 — Iconography: Bundled Qt Resource Set, Not External Library

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Extend the existing `icons.py` SVG factory** | The codebase already ships `icons.py` for icons used in queue/playlist/draft. Add the section-header set (keyboard, gear, layers, broadcast, palette) as additional factory functions. Same pattern, same registry. | **94/100** | **Selected** |
| External icon library (Material Symbols, Lucide, Phosphor) | Adds dependency, creates licensing review, requires bundle/manifest, font fallback risk. ~5 icons doesn't justify it. | 35/100 | Rejected |
| Embedded Unicode glyphs from a webfont | Visual inconsistency across themes (color-glyph emoji renders differently). Cannot tint to theme tokens. | 20/100 | Rejected |

**Why the existing SVG factory was selected:**

1. **Zero new dependencies.** `icons.py` already handles SVG generation, color tinting from theme tokens, and size variants.
2. **Theme-aware out of the box.** Icons tint to `theme.colors.gold` (or any theme token) automatically because the factory injects the color into SVG path fills.
3. **Pattern reuse.** Future icon needs (transcript history, sermon notes) extend the same factory.
4. **Known good.** The factory has been used since v0.7.6 (queue/playlist actions) and is stable.

### Decision 4 — Hover Affordances: Interactive Only

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Remove hover from non-interactive surfaces; reserve for buttons, links, and actionable cards** | Restores the UX contract that hover signals interactivity. Aligns with industry norms. | **96/100** | **Selected** |
| Keep hover on cards but only on click-through cards (queue items, history entries) | Same outcome as Decision 4 with extra rules. Less clear cascade. | 75/100 | Rejected |
| Keep hover everywhere as a "feedback" signal | Trains users to ignore hover. Antipattern. | 15/100 | Rejected |

**Implementation:** `QFrame[card="true"]:hover` rule is removed from `generate_stylesheet()`. The same rule on `QFrame[panel="true"]:hover` is reviewed — if the panel is non-interactive (most are), the hover rule is removed there too. Buttons, queue items, history entries keep hover.

**Long-term consequence:** Hover becomes a reserved signal for interactivity. Future panels that want hover feedback must opt in via an explicit `setProperty("interactive", True)` (or equivalent). This closes the silent-affordance regression that compounds every time a non-clickable card inherits hover by default. (R7)

### Decision 5 — Typography: Token-Based Scale, Not Selector-by-Selector

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Token-based scale in theme JSON `typography` section, generator emits selectors via loop** | Define the scale once in JSON. `generate_stylesheet()` iterates the section and emits one selector per entry. Adding a new size is a JSON edit. Themes can override per-scale (high contrast may want larger sizes for accessibility). | **91/100** | **Selected** |
| Two ad-hoc selectors hardcoded in `generate_stylesheet()` (original Decision 2 implementation) | Faster initial work. Re-creates the same anti-pattern v1.3.1 was designed to fix: every new size becomes a manual code edit + selector-name discussion. By Phase 4 the system is again inconsistent. | 50/100 | Superseded |
| Per-panel typography constants in each panel file | Same as the inline `setStyleSheet()` debt this stage is fixing. Trivially worst option. | 15/100 | Rejected |

**Why token-based was selected:**

1. **Decision 2's commitment is preserved, not abandoned.** Decision 2 chose a two-scale system (`compact`, `standard`). Decision 5 makes that choice extensible — the two scales become the initial population of the typography dict, not the entire universe of allowed sizes.
2. **Adding a new size is a data edit.** Phase 4 (theme designer) may want a `column-title` size larger than `standard`. v1.4.0 transcription will want a `caption` size for confidence scores. Both are JSON edits that the generator picks up automatically.
3. **Theme-overridable.** A high-contrast theme can declare `compact: 11px` for accessibility without affecting other themes. Schema v2 is already extensible — `typography` slots in alongside `colors`, `fonts`, `spacing`.
4. **The pattern survives every future panel.** v1.4.0 transcript history, v2.0.0 sermon notes, and any operator surface added beyond that consume the same scale.

**Long-term consequence:** After v1.3.1, `theme.py:generate_stylesheet()` no longer has hand-rolled typography selectors. Adding a new typography variant is a `theme.py` schema change, not a panel-by-panel edit. This closes the systemic version of the same problem the property-based QSS migration closes for component styling.

**Implementation consequence:** A new Phase 0 ("Typography System Foundation") is prepended to the original 4-phase plan. Phase 1 then consumes the typography system rather than declaring ad-hoc selectors. Phases 3 and 4 inherit the system. Total v1.3.1 duration moves from 2 weeks to 2.5 weeks (Phase 0 is 1 day; the remaining 0.4 day buffer accommodates Phase 3's 43-call migration surface).

---

### Decision 6 — Error Color: Promote to Theme Token

| Option | Summary | Score | Decision |
|--------|---------|------:|----------|
| **Add `red_text` token (red at 70% opacity) and `QLabel[error="true"]` selector** | Removes the last surviving inline `setStyleSheet()` from `settings_panel.py`. Mirrors `gold`/`green` token pairs already in every theme. | **90/100** | **Selected** |
| Reuse existing `red` token directly | Loses the 70% opacity treatment, which is intentional — full-strength red looks alarming on a status row that may show transient errors. | 50/100 | Rejected |
| Keep inline | Inconsistent with the rest of the migration. One exception breeds others. | 25/100 | Rejected |

**Implementation:** All 3 built-in themes get a `red_text` token (`rgba(<red components>, 0.7)`). New QSS rule `QLabel[error="true"] { color: {c("red_text")}; }`. NDI error label migrates from inline to `setProperty("error", True)`.

**Long-term consequence:** Status text colors (error, warning, success, info) become theme tokens, not inline literals. Future status surfaces — NDI sender errors, transcription confidence indicators, save-conflict warnings — consume `red_text`/`green_text`/etc. tokens that themes can override. The error label's 70% opacity treatment becomes the canonical "transient status" rendering across the app. (R7)

## 2a. Deviations from ROADMAP

Per `DOCUMENT_PROTOCOL.md` Rule 3, every sub-roadmap must document intentional deviations from `ROADMAP.md`. v1.3.1 has two: (B5)

### Deviation 1 — Schedule: 2 weeks → 2.5 weeks

- **ROADMAP says:** v1.3.1 is estimated at **2 weeks** (`ROADMAP.md` v1.3.1 row, prior to this sub-roadmap update).
- **Sub-roadmap says:** v1.3.1 is **2.5 weeks**, comprising Phase 0 (1 day) + Phases 1-4 (0.5 week each).
- **Justification:** Decision 5 (Typography: Token-Based Scale) added a 1-day infrastructure phase (Phase 0) prior to panel migration. The token-based system is necessary to prevent the per-panel migration in Phases 1-4 from re-creating the selector-by-selector anti-pattern this stage is meant to fix. The 0.5-day gap (Phase 0 = 1 day vs. the +0.5-week schedule slip) is reserved as buffer for Phase 3's 43-call home_panel.py migration surface, which is larger than initially projected. The ROADMAP version table has been updated to reflect 2.5 weeks; this deviation is now closed at the ROADMAP level but documented here for audit traceability.

### Deviation 2 — Theme schema additive `typography` section without `schema_version` bump

- **ROADMAP / v1.3.0 sub-roadmap §4.3 say:** Schema v2 themes have a fixed structure with sections `colors`, `fonts`, `spacing`, `animation`, `lower_third`, `fullscreen`. The v1.3.0 sub-roadmap §4.3 frames schema_version bumps as the mechanism for evolving the schema.
- **Sub-roadmap says:** Phase 0 adds a top-level `typography` section to all built-in theme JSONs **without** bumping `schema_version` from 2.0 to 2.1.
- **Justification:** v1.3.0's schema migration (`Theme._upgrade_to_v2`) explicitly populates **missing** sections with defaults (e.g., `fullscreen` was added this way for schema 1.0 themes). The same fallback pattern handles a missing `typography` section: `Theme.typography` defaults are emitted by `generate_stylesheet()` when a theme lacks the section. Old themes without `typography` continue to render correctly via these defaults, so no schema_version bump is required by the additive change. If a future change *removed* a token or *renamed* a section, that would warrant a bump. This deviation is non-breaking by construction.

## 3. Non-Regression & Scope Exclusions

These behaviors must remain stable:

- **Congregation display rendering is untouched.** v1.3.1 changes operator panel chrome only. `DisplayWidget.paintEvent`, font fitting, fullscreen/lower-third rendering, and theme application to the display continue to work exactly as in v1.3.0.
- **NDI capture pipeline is untouched.** `QWidget.grab()` captures whatever the operator panel renders, but operator panel changes don't affect the congregation display, and NDI streams the congregation display.
- **Theme schema accepts an additive `typography` section.** Phase 0 introduces a new top-level `typography` key in each built-in theme JSON. This is additive — themes without the section fall back to defaults emitted by `generate_stylesheet()`. No `schema_version` bump because v2 already permits additive top-level sections (per v1.3.0 sub-roadmap §4.3). User-created themes (e.g., `test_1.json`) without the section continue to render via the same fallback. (R4)
- **`red_text` token is added to all 3 built-in theme JSONs.** Additive, no schema bump. User-created themes without the token fall back to the underlying `red` token via QSS color cascade. (R4)
- **Theme Designer behavior is unchanged.** Property editor, preview, save/apply flow all continue to work.
- **Hotkey routing is unchanged.** All `Ctrl+Shift+*` shortcuts continue to target the active output channel.
- **Settings persistence is unchanged.** No new settings keys are introduced.
- **Per-channel theme application is unchanged.** Each channel's `DisplayWidget.set_theme()` continues to apply per-channel themes without bleeding into operator panel chrome.

**Scope boundaries:**

- v1.3.1 modifies the four primary operator surfaces: `home_panel.py`, `settings_panel.py`, `theme_designer.py`, and the dialogs spawned from settings (`_test_hotkeys`, `_show_full_details`).
- v1.3.1 does NOT modify: NDI files, channel manager, display facade, display window, display widget, display core, display channel, settings manager, document manager, queue/playlist models, or any database layer.
- Any layout change in operator panels must preserve existing widget IDs and signal connections.

## 4. Phases

### Phase 0 — Typography System Foundation (1 day)

**Objective:** Establish the token-based typography scale (Decision 5) before any panel migration begins. After Phase 0, every subsequent phase consumes typography variants by name rather than declaring ad-hoc selectors. Settings panel is not touched yet — that's Phase 1.

**Deliverables:**

- New top-level `typography` section in all 3 built-in theme JSONs (`dark_gold`, `light`, `high_contrast`). Initial population: 4 named scales.

  ```json
  "typography": {
    "compact":   { "size": 9,  "weight": "Bold",   "uppercase": true,  "letter_spacing": 2 },
    "standard":  { "size": 13, "weight": "Bold",   "uppercase": false, "letter_spacing": 0 },
    "body":      { "size": 10, "weight": "Normal", "uppercase": false, "letter_spacing": 0 },
    "hint":      { "size": 9,  "weight": "Normal", "uppercase": false, "letter_spacing": 0 }
  }
  ```

  Four entries selected to cover known consumers: section headers (compact + standard), body labels, and hint labels. **A `caption` entry is intentionally deferred to v1.4.0** when transcription confidence scores create a real consumer; adding it now with values identical to `hint` would create implementation ambiguity (developers couldn't visually distinguish which to use). When v1.4.0 lands, `caption` is added as a JSON edit — exactly the workflow Decision 5 enables. (R5)

- `theme.py` extended to parse the `typography` section into `Theme.typography: dict[str, dict]`. Falls back to a default scale when a theme lacks the section.

- `generate_stylesheet()` extended to iterate `Theme.typography` and emit one QSS rule per entry:

  ```python
  for name, spec in theme.typography.items():
      rules.append(f'QLabel[typography="{name}"] {{ ... }}')
      rules.append(f'QLabel[section-header="{name}"] {{ ... }}')
  ```

  Both `[typography="..."]` and `[section-header="..."]` selectors are emitted for every entry. The dual emission preserves backward compatibility with the v1.3.0 `section-header` property while introducing `typography` as the forward-looking property name. Phases 1, 3, 4 use whichever name fits semantically per call site.

- The existing v1.3.0 `QLabel[section-header="true"]` selector is **kept as a deprecated alias** that emits the same QSS as the new `compact` variant. This eliminates any transient broken state between Phase 0 and Phase 1. Phase 1 migrates the single call site (`_make_section_header` in `settings_panel.py`) from `setProperty("section-header", True)` to `setProperty("section-header", "compact")`. Phase 1's DoD removes the deprecated alias once the migration is verified.

- **`Theme.deep_copy()` (theme.py:165-178) extended to include `typography`** in its reconstruction dict. Without this, the Theme Designer's deep-copied editing sandbox loses typography on every edit cycle. Mirror the pattern used for `colors`/`fonts`/`spacing`/`lower_third`/`fullscreen`: `"typography": copy.deepcopy(self.typography)`.

- **`Theme._to_dict()` (theme.py:184-197) extended to include `typography`** in its serialization output. Without this, `Theme.save()` and `Theme.save_as()` silently strip the typography section from disk. Insert `"typography": self.typography` between `spacing` and `fullscreen` (preserves the established section order in JSON files).

- Guard script `pre_commit_checks.py` extended: verify all built-in themes (the 3 in `BUILTIN_THEME_IDS` — `dark_gold`, `light`, `high_contrast`) have a `typography` section with the 5 expected keys, and that `theme.py` emits typography selectors. (R4: scope to `BUILTIN_THEME_IDS` rather than counting JSON files, since user-created themes like `test_1.json` are deletable.)

**Definition of Done:**

- [ ] All 3 built-in theme JSONs (`dark_gold`, `light`, `high_contrast`) have a `typography` section with 4 named scales (compact, standard, body, hint). User-created themes lacking the section fall back to the defaults emitted by `generate_stylesheet()`. (R4, R5: scope to built-ins; defer `caption` to v1.4.0.)
- [ ] `Theme.typography` attribute parses and exposes the dict.
- [ ] **`Theme.deep_copy()` includes `typography` in its reconstruction dict.** (B1)
- [ ] **`Theme._to_dict()` includes `typography` in its serialization output.** (B1)
- [ ] Schema v1.0 themes (none currently exist; safety check) get a default typography dict via `_upgrade_to_v2()`.
- [ ] `generate_stylesheet()` emits `QLabel[typography="X"]` and `QLabel[section-header="X"]` for every X in the dict.
- [ ] The legacy `QLabel[section-header="true"]` selector is **kept as a deprecated alias** of `compact`. (It is removed in Phase 1's DoD after the single call site is migrated.)
- [ ] All 4 themes render existing panels (home, settings, theme designer) **without any visual regression**. The deprecated alias guarantees the v1.3.0 settings panel section headers continue rendering correctly until Phase 1 explicitly migrates them.
- [ ] Round-trip test: open a theme in the Theme Designer, edit a property, save. Verify the saved JSON file retains its `typography` section. (Validates B1 fix end-to-end.)
- [ ] Guard scripts pass.

**Phase 0 hand-off note:** Phase 0 introduces no visual change. Existing panels render exactly as they did in v1.3.0. The new typography selectors are dormant until consumers opt in via `setProperty("typography", "X")` or the migration in Phase 1. This is by design — Phase 0 ships infrastructure, not user-visible work.
